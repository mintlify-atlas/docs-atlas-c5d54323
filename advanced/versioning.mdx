title: "Versioning @types packages in DefinitelyTyped"
description: "Learn how DefinitelyTyped package versions relate to library versions, how to handle major version bumps, and how to maintain multiple versions simultaneously."
---

DefinitelyTyped packages are versioned to match the library they type. Understanding the versioning scheme helps you know which `@types` version to install and how to handle library upgrades.

## How versioning works

Each `@types` package version follows the pattern `major.minor.patch`, where:

- `major.minor` matches the version of the underlying library
- `patch` is incremented each time the `@types` package is updated for the same library version

The `package.json` in DefinitelyTyped always uses `major.minor.9999` — the `.9999` suffix ensures local packages are treated as newest during development. The actual patch version is managed by the publishing toolchain.

For example, `@types/node` for Node 20.8.x:

```json package.json
{
    "name": "@types/node",
    "version": "20.8.9999"
}
```

This publishes as `@types/node@20.8.x` on npm.

## Updating to a new minor version

When a library releases a new minor version with new APIs, update the `version` field in `package.json` to match:

```json
{
    "version": "1.3.9999"
}
```

<Warning>
Always update `major.minor` in `package.json` when adding types for new library features. If you don't, the version numbers go out of sync and `npm update` may not work correctly for users.
</Warning>

## Handling major version breaking changes

When a library releases a new major version with breaking changes, the DefinitelyTyped package must also be updated. Users on the old major version still need their types, so you maintain both simultaneously.

### Create a versioned subfolder

Copy the current types into a versioned subfolder before making breaking changes:

```
types/history/
  index.d.ts          ← updated v4 types
  package.json        ← version: "4.0.9999"
  v2/
    index.d.ts        ← preserved v2 types
    package.json      ← version: "2.4.9999"
```

The subfolder's `package.json`:

```json types/history/v2/package.json
{
    "private": true,
    "name": "@types/history",
    "version": "2.4.9999"
}
```

### Depending on a specific version

Other packages that need the old version can specify it in their dependencies:

```json
{
    "dependencies": {
        "@types/history": "^2"
    }
}
```

<Note>
Use `import` (not triple-slash `reference types` directives) when depending on versioned packages, as triple-slash references don't work with path mapping.
</Note>

## Version lag

It's common for `@types` packages to lag behind library releases by days, weeks, or even months — library users (not maintainers) often update DefinitelyTyped when new features are released. If you're affected by a lag, consider submitting a PR yourself.

## Breaking type changes within a version

When `@types` packages track a library version closely, all changes (breaking or not) are published as patch revisions. DT maintainers weigh the popularity of the package, the upside of the change, and whether it can wait for a major version bump before approving breaking type changes.
